# PMForge QA Review - 15 June 2026

Source: the screenshots and notes in *PM Forge Test 15-June-2026*. The app
launches and renders a polished dark UI; every issue below is a functional
bug or a UX gap, not a startup failure. Almost all of the reported errors
reduce to **two root causes**.

## Root cause A - empty dates can't be unmarshaled into `time.Time` (HIGH, pervasive)

**Reported as**

- "Save failed: error parsing arguments: parsing time \"\" as \"2006-01-02T15:04:05Z07:00\"" when saving Stakeholders.
- The same error when saving Project Settings ("Unable to save project settings"), which also blocks theme changes ("Unable to change themes, however able to change between two fonts").
- "Unable to select, view, or use these New Charts" - creating a chart hits the same failure.

**Why.** Several Wails-bound methods take a struct argument whose
`CreatedAt`/`UpdatedAt` fields are `time.Time`:

- `App.SaveChart(db.Chart)` - `db.Chart.CreatedAt/UpdatedAt time.Time`
- `App.SaveStakeholder(db.Stakeholder)` - same
- `App.UpdateProjectMeta(db.Project)` - same (Project Settings save)
- `App.SaveDocument(db.Document)` - same

Wails unmarshals JS arguments into Go structs and parses `time.Time` with
RFC3339. The frontend sends an **empty string** for these fields on new or
edited records. The smoking gun is `Dashboard.svelte` `newChart()`, which
sends `created_at: '', updated_at: ''`; Wails then fails with
`parsing time "" as "2006-01-02T15:04:05Z07:00"` and the whole call is
rejected before the Go body runs. Fonts still work because the font change
goes through `App.SetDefaultFont(string)` (a plain string), and themes fail
because they ride along with the project/settings save that carries the
`time.Time` fields.

**Fix (recommended).** Change `CreatedAt`/`UpdatedAt` from `time.Time` to
`string` (RFC3339 text) on `db.Chart`, `db.Document`, `db.Stakeholder`, and
`db.Project`, and adjust their SQLite scan/format code accordingly. This
matches the project's own conventions - `db.Project.StartDate`/`EndDate` are
already `string`, and the agile package stores timestamps as strings on the
wire. An empty string is then a valid value, and these are server-managed
fields the client never needs to parse anyway. This one change unblocks
stakeholders, project settings, themes, charts, and documents.

## Root cause B - multi-return methods resolve to `null` and break destructuring (HIGH)

**Reported as**

- "Create failed: TypeError: null is not an object (evaluating 'window.go.main.App.CreateProjectFromLaunchpad(...)')" - **but the project is still created** (Test Prince Project-2 through -9 all exist).
- "Could not load backlog: TypeError: null is not an object (evaluating 'window.go.main.App.EnsureDefaultBoard()')".

**Why.** These methods return multiple values:

- `CreateProjectFromLaunchpad(...) (db.Project, []templates.SeedReceipt, string, error)`
- `EnsureDefaultBoard() (agile.Board, []agile.Column, error)`

and the frontend destructures an array from the result:

- `const [project, , projectPath] = await ...CreateProjectFromLaunchpad(...)`
- `const [b, cols] = await ...EnsureDefaultBoard()`

The Go call **does** execute (hence the project is created), but the awaited
result comes back `null`, and destructuring `null` throws
`null is not an object (evaluating '<the right-hand call>')` in WebKit - which
is exactly the wording shown. So `window.go.main.App` is present and callable;
only the multi-value result is mis-marshaled.

**Fix (recommended, guaranteed).** Have these methods return a **single
struct** instead of multiple values, and read fields on the frontend:

- `CreateProjectFromLaunchpad` -> `LaunchpadResult{ Project, Seeds, Path }`; frontend uses `res.project` / `res.path`.
- `EnsureDefaultBoard` -> `BoardWithColumns{ Board, Columns }`; frontend uses `res.board` / `res.columns`.

This removes any dependence on Wails multi-return array marshaling.

**Also.** Building via the proper Wails CLI (`wails build` with binding
generation) may resolve this on its own, because the screenshots were taken
from the earlier hand-rolled `go build` that never ran Wails' binding step.
The `-skipbindings` flag I added earlier was only a workaround for the
codesign error, which `scripts/wails-build.sh` now handles by stripping
extended attributes and re-signing. **Recommendation: drop `-skipbindings`
so bindings are generated normally, then re-test B before deciding whether
the struct-return refactor is still needed.** The single-struct refactor is
the safe, marshaling-independent guarantee.

## Other functional issue

### Certificate cannot be chosen or imported from the signing modal (MEDIUM)

The Digital Signature dialog shows "(no certificate configured)" with no way
to browse for or import a certificate. `App.ChooseCertFile()` exists and is
used by `SignatureSettings.svelte`, but the `SignCertificateModal` doesn't
expose it. **Fix:** add a "Choose certificate..." button to the modal that
calls `ChooseCertFile`, or direct users to Project Settings -> Signature.
Note that persisting the choice depends on Root cause A being fixed (the
settings save path).

## UI/UX gaps (feature requests from the notes)

1. **Your Projects: Delete and Clone.** Add per-row Delete (reuse the
   Dashboard's existing two-step confirm pattern) and Clone actions. Today
   there is no way to remove or duplicate a project from the list.
2. **Global application settings.** Add an app-level Settings entry distinct
   from per-project settings (default font, theme defaults, update channel,
   data location).
3. **Native menu bar (Open / Help / About / Settings).** Wails supports a
   real application menu via `options.App.Menu` in the root `main.go`; there
   is currently none. Add File (Open/New/Close), Help (About, docs link),
   and Settings items.
4. **"Available Chart Templates" is read-only.** The panel lists 21 templates
   grouped by engine but the entries aren't clickable. Either make each entry
   create that chart (preferred) or visually mark the panel as informational
   so it doesn't read as broken.
5. **Theme switching** is blocked by Root cause A; verify the theme dropdown
   persists once settings save works.
